Module 2: Lambda IAM

Overview​

Lambda functions are an important part of serverless architecture. You can use AWS Identity and Access Management (IAM) to manage access to Lambda and resources such as functions and layers.

Every Lambda function has an IAM role called an execution role. In this role, you can attach a policy that defines the permissions your function needs to access other AWS services and resources.

Module 2A: Scoped Down Permission using IAM Access Analyzer​

  1. Open the Lambda Analytics function code in src/app/customUnicornAnalytics.js

  1. Open the AWS Lambda Console and search for CustomUnicornAnalyticsFunction

  1. Click on the function to open its detail page

  1. Go to Configuration > Permissions tab. Click on the Execution role name to open the IAM role

  2. Review the attached permissions (note: currently has AdministratorAccess)

  1. Scroll down to "Generate policy based on CloudTrail events" and click Generate Policy

  2. On the Generate policy screen:

    • Time period: Select "Last 1 day"
    • Region: Select "US East (N. Virginia)"
    • CloudTrail Trail: Select "Serverless-Security-Workshop-Trail"
    • Service role: Select "Create and use a new service role"

  1. Click Generate Policy and wait for completion

  2. Wait for the status to show Success (may take a few minutes)

  1. Click View generated policy

  2. Review the generated policy showing DynamoDB and Secrets Manager actions

  3. Select the appropriate actions:

    • DynamoDB: Select PutItem
    • Secrets Manager: Select GetSecretValue (if Module 4 was completed)

  1. Click Next to proceed to customization

  2. On the Customize Permissions page, remove unnecessary policy statements (keep only DynamoDB and Secrets Manager)

  3. Set the Resource ARN for DynamoDB table and Secrets Manager secret

  1. Click Next to continue

  2. Enter policy name: CustomUnicornAnalyticsScopedDownPolicy

  1. Click Create and attach policy

  1. Back on the IAM role page, remove the AdministratorAccess policy by clicking Remove

  1. Verify in the Lambda console that the execution role is updated

  1. Navigate to the DynamoDB Console and verify new records are being inserted into CustomizeUnicorns-CustomizationDemandAnalytics table

Module 2B: Attribute Based Access Control (ABAC)​

Part 1: Explore the Infrastructure​

  1. Open the AWS IAM Console

  2. Click on Users under Access Management in the left navigation

  3. Search for and open serverless-dev-user

  1. Review the attached policy allowing sts:AssumeRole and sts:TagSession

  1. Click on Roles in the left navigation. Search for ServerlessABACDemoRole and open it

  1. Review the attached managed policies:
    • ServerlessABACDemoPolicyLambdaInvoke (GetFunction, InvokeFunction)
    • ServerlessABACDemoPolicyListLambdaIAM (List and Get permissions)

  1. Copy the "Link to switch roles in console" URL from the top right (for later use)

Part 2: Log In as Developer User and Test Function Invoke​

  1. Open a new Incognito/Private window and navigate to your AWS account login page

    https://ACCOUNT_ID.signin.aws.amazon.com/console/

  2. Log in with developer credentials:

    • IAM user name: serverless-dev-user
    • Password: pA$$woRd1

  1. After login, paste the Switch Role URL copied earlier

  2. Verify you've switched to ServerlessABACDemoRole

  1. Open the Lambda Console and search for CustomUnicornAnalyticsFunction

  1. Click on the function to open it

  2. Go to Configuration > Tags tab

  3. Verify the function is tagged with application: customizeUnicorn

  1. Click on the Test tab and click Test button to invoke the function

  2. Verify the function executes successfully

Part 3: Modify Permission to Enforce Tag-Based Access​

  1. Go back to your original browser window (logged in as IAM Admin)

  2. Open ServerlessABACDemoRole and go to Permissions policies

  3. Expand ServerlessABACDemoPolicyLambdaInvoke and click Edit

  1. Add the following condition to the policy:

    "Condition": {
      "StringEquals": {
        "aws:ResourceTag/application": "${aws:PrincipalTag/application}"
      }
    }

  1. Click Next to review changes

  1. Click Save Changes

Part 4: Verify Developer User Cannot Invoke Function​

  1. Go back to the developer user's browser window

  2. In the Lambda Console, click Test button again to try invoking the function

  3. Verify the execution fails with an access denied error

Part 5: Tag Developer Role with Matching Tag​

  1. Go back to the IAM Admin browser window

  2. Open ServerlessABACDemoRole and click on the Tags tab

  3. Click Add new tag

  1. Add tag:
    • Key: application
    • Value: customizeUnicorn

  1. Click Save changes

Part 6: Verify Developer Can Now Invoke Function​

  1. Go back to the developer user's browser window

  2. In the Lambda Console, click Test button to invoke the function

  3. Verify the function now executes successfully

Module 2C: Permission Boundary​

Part 1: Modify Developer Role to Allow IAM Role Creation​

  1. Open the AWS IAM Console

  2. Click on Policies under Access Management in the left navigation

  3. Search for and open ServerlessPermissionBoundaryDemoIAMActionsPolicy

  1. Review the policy allowing CreateRole, DeleteRole, and other IAM actions

  2. Note that the resource (IAM Role) name is enforced to start with Serverless-

  1. Click on Roles in the left navigation

  2. Search for and open ServerlessABACDemoRole

  1. Click Add permissions > Attach policies

  2. Search for ServerlessPermissionBoundaryDemoIAMActionsPolicy

  3. Check the policy and click Add permission

  1. Leave this browser window open as you will return to modify permissions later

Part 2: Test Developer User Can Create IAM Roles​

  1. Open your developer user browser (ServerlessABACDemoRole session)

  2. Open the AWS IAM Console

  3. Click Roles under Access management in the left navigation

  4. Click Create role button

  1. On Select trusted entity page, select Trusted entity type as AWS Service

  2. Select Use case as Lambda

  3. Click Next button

  1. On Attach permissions page, search for AWSLambdaExecute

  2. Select AWSLambdaExecute policy and click Next

  1. On Name, review, and create page, enter role name: Serverless-CustomAnalyticsExecutionRole

  2. Click Create role

  1. Verify the role was created successfully

  2. Open the AWS Lambda Console

  3. Click Create function button

  1. Enter function name: ServerlessUnicornCustomFunction

  2. Select Runtime as your preferred Node.js version

  3. Under Execution role, select Use an existing role

  4. Select the role you just created: Serverless-CustomAnalyticsExecutionRole

  5. Click Create function

  1. Verify the function was created successfully
  2. Leave this browser window open for later testing

Part 3: Review and Apply Permission Boundary Policy​

  1. Go back to the IAM Admin browser window (WSParticipantRole session)

  2. Open the AWS IAM Console

  3. Click on Policies in the left navigation

  4. Search for ServerlessPermissionBoundaryDemoBoundaryPolicy

  5. Click to open the policy and review it

  1. Note this policy allows limited operations on DynamoDB, S3, and CloudWatch Logs

  2. Copy the ARN of this policy (shown at the top right) to your notes

  3. Click on Policies again in the left navigation

  4. Search for ServerlessPermissionBoundaryDemoIAMActionsPolicy

  5. Click to open and click Edit policy button

  1. Add the following boundary condition to the policy:

    "Condition": {
      "ArnEquals": {
        "iam:PermissionsBoundary": "arn:aws:iam::YOUR_ACCOUNT_ID:policy/ServerlessPermissionBoundaryDemoBoundaryPolicy"
      }
    }

  1. Click Next to review changes

  2. Click Save changes

Part 4: Verify Developer Must Specify Permission Boundary​

  1. Go back to the developer user browser (ServerlessABACDemoRole session)

  2. Open the AWS IAM Console

  3. Click Roles under Access management in the left navigation

  4. Click Create role button

  5. On Select trusted entity page, select Trusted entity type as AWS Service

  6. Select Use case as Lambda

  7. Click Next button

  8. On Attach permissions page, search for AdministratorAccess

  9. Select AdministratorAccess policy and click Next

  1. On Name, review, and create page, enter role name: Serverless-CustomizeUnicornDemandRole

  2. Click on Set permissions boundary section

  3. Check Use a permissions boundary to control the maximum role permissions

  4. Search for and select ServerlessPermissionBoundaryDemoBoundaryPolicy

  1. Click Create role

  2. Verify the role was created successfully with the permission boundary

Part 5: Create Lambda Function with Boundary-Constrained Role​

  1. Open the AWS Lambda Console

  2. Click Create function button

  3. Enter function name: ServerlessUnicornCustomAnalyticsFunction

  4. Select Runtime as your preferred Node.js version

  5. Under Execution role, select Use an existing role

  6. Select the role you just created: Serverless-CustomizeUnicornDemandRole

  7. Click Create function

  8. Verify the function was created successfully

  9. Confirm that the permission boundary successfully limited the developer's permissions while allowing role creation
